# PRD: AI 游戏创作学习平台 MVP

## 1. 产品名称

暂定名: **PlayCraft Lab / 创游实验室**

备选名称:

- VibeGame Lab
- PromptPlay
- GameCraft Studio
- 想象工坊
- AI 创游学堂
- 游戏表达实验室

## 2. 产品定位

PlayCraft Lab 是一个通过游戏创作学习 Vibe Coding 的平台。

用户不需要先学习代码, 而是通过描述游戏想法、拆解游戏规则、与 AI 对话、试玩结果并持续修改, 学习如何把一个模糊创意变成可运行的互动作品。

核心定位:

> 用做小游戏的方式, 学习如何和 AI 协作创作。

一句话介绍:

> 从一句游戏想法开始, 学会表达、拆解、试玩和迭代, 和 AI 一起做出可玩的小游戏。

## 3. 产品愿景

AI 时代, 创作能力不只来自会不会写代码, 而来自:

- 能不能把想法说清楚
- 能不能把复杂目标拆成小任务
- 能不能定义规则和边界
- 能不能判断 AI 输出是否符合预期
- 能不能通过反馈持续改进结果

PlayCraft Lab 希望通过“做游戏”这个直观、有趣、可反馈的场景, 帮助用户学习这些能力。

## 4. MVP 目标

MVP 的目标不是做一个完整游戏引擎, 也不是做一个儿童教育平台, 而是验证:

> 用户是否可以通过一个结构化学习流程, 完成一次完整的 Vibe Coding 游戏创作体验。

MVP 成功标准:

1. 用户能从一句模糊想法开始, 逐步补全为清晰的游戏设计。
2. 用户能通过 AI 生成一个简单可玩的 2D 小游戏。
3. 用户能试玩游戏, 并用自然语言提出修改。
4. 用户能理解自己修改的是哪类游戏概念。
5. 用户最终能得到一个作品和一张学习复盘卡。

## 5. 目标用户

### 5.1 零基础创作者

没有编程经验, 但有游戏或互动作品想法的人。

需求:

- 不想先学代码
- 想快速做出可玩的东西
- 需要 AI 引导自己表达清楚

### 5.2 学生和青少年

对游戏感兴趣, 希望通过创作学习 AI 工具使用和创意思维。

需求:

- 体验要有趣
- 反馈要即时
- 学习过程不能太抽象
- 希望能保存和展示作品

### 5.3 成人初学者

想了解 Vibe Coding, 但缺少练习场景。

需求:

- 学会如何和 AI 沟通
- 学会拆解需求
- 学会通过试玩反馈迭代
- 希望看到创作过程结构化

### 5.4 老师/家长/教育者

希望用游戏创作作为学习项目。

需求:

- 学习目标清晰
- 过程可观察
- 作品可复盘
- 内容相对安全可控

## 6. 非目标用户

MVP 暂不服务以下用户:

- 专业游戏开发者
- 需要完整商业游戏引擎的人
- 想自由编写复杂代码的人
- 需要 3D 游戏制作的人
- 需要多人联网游戏的人
- 需要发布到 App Store / Steam 的用户

## 10. MVP 游戏类型

第一版只支持一种基础游戏模型:

> 2D 收集 + 躲避类游戏

游戏结构:

```text
玩家控制一个主角
主角在场景中移动
收集目标物品获得分数
躲避障碍或敌人
达到目标分数胜利
生命值归零或时间结束失败
```

可表达为不同主题:

```text
小猫收星星, 躲开怪物
飞船收能量球, 躲开陨石
机器人找电池, 躲开巡逻机器人
小鱼收珍珠, 躲开水母
骑士找宝石, 躲开幽灵
```

## 11. 用户流程

### 11.1 新用户首次体验流程

1. 用户进入首页
2. 点击 `开始学习 Vibe Coding`
3. 进入学习路径: `从一句话到一个小游戏`
4. 用户输入游戏想法:

```text
我想做一个飞船收集能量的游戏。
```

5. AI 引导补充:

```text
飞船在哪里?
A. 太空
B. 海底
C. 城市上空
D. 自己写一个
```

6. 用户选择: 太空
7. AI 继续引导:

```text
飞船要躲开什么?
A. 陨石
B. 黑洞
C. 敌人飞船
D. 自己写一个
```

8. 用户选择: 陨石
9. AI 整理游戏结构:

```text
主角: 飞船
场景: 太空
目标: 收集能量球
障碍: 陨石
胜利: 收集 10 个能量球
失败: 碰到陨石 3 次
```

10. 用户确认
11. 系统生成游戏
12. 用户试玩
13. 用户输入:

```text
陨石太慢了, 让它们快一点。
```

14. AI 修改游戏
15. 系统提示:

```text
你刚刚调整了游戏难度。
```

16. 用户保存作品
17. 系统生成复盘卡

## 13. Agent 设计

### 13.1 Agent 角色

Agent 是一个 **Vibe Coding 学习助手**。

它不是单纯执行命令, 而是帮助用户:

- 说清楚想法
- 拆解游戏功能
- 理解游戏概念
- 提出修改建议
- 完成试玩后的迭代
- 复盘学习过程

### 13.2 Agent 语气

原则:

- 简洁
- 友好
- 不讲代码
- 少用术语
- 多给选择
- 一次只问一个关键问题
- 修改后解释学习点

示例:

```text
我明白了, 你想做一个飞船收集能量的游戏。
现在还差一个挑战: 玩家需要躲开什么?
```

### 13.3 Agent 引导流程

Agent 按以下顺序补全游戏:

1. 主角
2. 场景
3. 目标
4. 收集物
5. 障碍
6. 规则
7. 胜利条件
8. 失败条件
9. 难度

### 13.4 Agent 修改能力

MVP 支持以下修改:

```text
更换主角
更换场景
更换收集物
更换障碍
调整收集物数量
调整障碍数量
调整障碍速度
调整玩家速度
调整胜利分数
调整生命值
调整失败条件
调整游戏难度
```

### 13.5 Agent 安全规则

MVP 虽不定位儿童产品, 但仍应保持基础内容安全。

Agent 应避免生成:

- 血腥暴力
- 成人内容
- 仇恨内容
- 自残内容
- 真实个人信息请求
- 违法行为指导
- 恐吓或骚扰内容

当用户输入不合适内容时, Agent 应改写为安全版本。

示例:

```text
这个设定可以改成更适合游戏创作的版本:
怪物碰到主角时, 主角失去一颗生命并回到起点。
```

## 19. MVP 验收标准

MVP 完成时应满足:

1. 用户能输入一个游戏想法。
2. AI 能引导用户补全至少 5 个游戏结构项。
3. 系统能生成一个可玩的 2D 收集/躲避类游戏。
4. 用户能通过自然语言完成至少 5 种修改。
5. 每次修改能更新游戏配置并体现在试玩区。
6. 系统能记录修改日志。
7. 系统能生成作品复盘卡。
8. 用户能保存并重新打开作品。
9. 用户不需要看到或编辑代码。
10. 平台能解释用户每一步练习的 Vibe Coding 能力。

## 21. 关键产品原则

1. **学习优先, 不是生成优先**

   AI 不只是帮用户做完, 而是帮助用户理解怎么表达、怎么拆解、怎么修改。

2. **不从代码开始**

   MVP 不展示代码, 不要求用户理解语法。

3. **游戏是学习载体**

   游戏让用户获得即时反馈, 帮助他们理解规则和迭代。

4. **每一步都要有可见结果**

   用户说出想法后, 应尽快看到游戏变化。

5. **每次修改都要对应一个学习点**

   例如难度、目标、规则、反馈、胜利条件。

6. **先限制游戏类型, 做好体验**

   第一版只支持收集/躲避类游戏, 避免系统过度复杂。

7. **为儿童版和课堂版预留能力**

   即使 MVP 是通用版, 也要在素材、安全、权限、复盘数据上预留扩展空间。

## 22. MVP 推荐开发顺序

1. 定义游戏配置 Schema
